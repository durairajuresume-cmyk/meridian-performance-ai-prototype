# Meridian Performance AI Workspace

A functional prototype built for a Product Manager take-home assignment. It demonstrates the
**Meridian-specific Q1 pilot configuration** running on top of Performance AI's existing platform
capabilities (connectors, full-funnel view, AI diagnostics, dashboards, alerts, playbooks, deep
research) for **Meridian Capital Finance**, a large Indian NBFC/BFSI lender evaluating Performance
AI for acquisition marketing.

This is **not** a rebuild of the generic Performance AI platform. It is a Meridian/BFSI pilot
layer: offline loan disbursal attribution, a CRM upload bridge while Microsoft Dynamics migration
is pending, evidence-backed AI diagnostics with confidence scoring, a Microsoft Teams alert
workflow, recommendation-only frequency intelligence, and BFSI-specific AI guardrails.

## What this prototype demonstrates

The core Q1 pilot workflow, end to end:

1. **View trusted full-funnel campaign performance** — Overview dashboard with KPIs, connected
   source status, and a campaign performance table.
2. **Ask AI why campaign performance changed** — AI Copilot with suggested prompts and free-text
   input, grounded in connected Meridian data.
3. **See an evidence-backed recommendation with confidence** — every AI response cites specific
   evidence, a recommendation, and a confidence score, plus a guardrail disclosure.
4. **Drill into a campaign funnel** — clicking any campaign (or "View Evidence" on an AI response)
   opens a drilldown drawer with funnel, offline attribution, and root-cause signals.
5. **Send an insight to Microsoft Teams** — "Send to Teams" opens a preview modal; sending shows a
   confirmation toast.

Supporting screens: **Campaigns** (full table with status filters), **Reports** (Q1 pilot reports
with previewable summaries), **Alerts** (configured alert rules routed to Teams), and
**Governance** (BFSI guardrails plus a worked example of the AI declining an out-of-scope prompt).

## What is mocked

- **All data is static**, defined in `lib/mock-data.ts` and `lib/ai-responses.ts`. There are no
  real API calls, no database, and no environment variables required.
- **AI responses** are pre-written and selected by matching the submitted prompt text (the four
  suggested chips return detailed, hand-authored diagnostics; any other prompt returns a grounded
  fallback summary; a small set of off-topic keywords — e.g. "IPL", "sponsorship" — triggers the
  guardrail decline response shown in Governance). There is no live LLM call.
- **"Send to Teams"** only shows a preview modal and a local success toast — no message is
  actually sent to Microsoft Teams.
- **Connector statuses** (Google Ads, Meta, DV360, CRM Upload, Microsoft Dynamics, etc.) are fixed
  mock values, not live integration health checks.
- **"Last refreshed N seconds ago"** in the header ticks locally in the browser; it does not
  reflect a real data refresh.

## What would be real in production

- Connector status would come from Performance AI's existing connector health/auth layer (Google
  Ads, Meta, Snapchat, TikTok, DV360, App Analytics) plus the Meridian-specific CRM upload bridge
  and, eventually, a live Microsoft Dynamics integration.
- Offline loan disbursal attribution would run through Performance AI's offline conversion
  matching pipeline against Meridian's CRM exports, with match-rate and reconciliation-lag
  reporting.
- AI diagnostics would be generated by Performance AI's existing diagnostics/insights engine,
  constrained to Meridian's connected data sources and BFSI guardrail policy, with evidence
  citations pulled from real metrics and a calibrated confidence model.
- "Send to Teams" would call Performance AI's existing Teams alert workflow/connector (the same
  mechanism used for the configured alert rules) via Microsoft Graph/Teams webhook.
- Frequency intelligence would read live frequency metrics from each platform; capping would
  remain **recommendation-only** until Meridian explicitly grants platform write-access for
  automated execution — this prototype intentionally does not present auto-capping as built.
- Governance guardrails (Meridian context scoping, evidence/confidence requirements, PII masking,
  advisory-only mode, BFSI claim safety, audit logging) would be enforced server-side as policy on
  top of the diagnostics engine, not just rendered as UI copy.

## Tech stack

- Next.js 16 (App Router) + TypeScript
- Tailwind CSS v4
- shadcn/ui-style component primitives (`components/ui/*`) built with `class-variance-authority`,
  `clsx`, `tailwind-merge`
- `lucide-react` for icons
- Mock data only — no real API keys, no environment variables

## Project structure

```
app/                  Routes: Overview (/), Campaigns, AI Copilot, Reports, Alerts, Governance
components/
  layout/              Sidebar, header, app shell
  providers/           Workspace context (drawer / Teams modal / toast state)
  overlays/             Campaign drilldown drawer, Teams alert modal, toast
  dashboard/             KPI cards, connected sources, campaign table, frequency intelligence
  copilot/                Prompt chips, AI response card
  reports/, alerts/, governance/   Per-section cards
  ui/                       Reusable primitives: button, card, badge, input, progress
lib/
  types.ts             Shared TypeScript types
  mock-data.ts          All mock KPIs, campaigns, sources, reports, alerts, guardrails
  ai-responses.ts        Canned AI response generator + guardrail decline logic
  utils.ts                `cn()` class-merging helper
```

## How to run locally

Requires Node.js 18.18+ (Node 20+ recommended).

```bash
npm install
npm run dev
```

Open [http://localhost:3000](http://localhost:3000). No environment variables or API keys are
needed.

Other useful commands:

```bash
npm run build   # production build
npm run start   # serve the production build
npm run lint    # eslint
```

## How to deploy to Vercel

1. Push this repository to GitHub (or another Git provider Vercel supports).
2. In Vercel, choose **Add New Project** and import the repository.
3. Framework preset: **Next.js** (auto-detected). No environment variables are required.
4. Deploy — the default build (`npm run build`) and output settings work as-is.

Alternatively, from the project root with the [Vercel CLI](https://vercel.com/docs/cli) installed:

```bash
vercel
```

## Product rationale

This prototype does not rebuild the generic Performance AI platform. It demonstrates the
Meridian-specific Q1 pilot layer on top of existing Performance AI capabilities: BFSI offline
attribution, the CRM upload bridge, evidence-backed AI diagnostics, the Teams workflow, frequency
intelligence, and enterprise guardrails. The goal is to show a CMO, Performance Lead, and Martech
Head at a BFSI lender exactly what changes — and what guardrails apply — when Performance AI is
configured for their environment, without implying that any of this is generic, unproven, or
built from scratch for Meridian.
